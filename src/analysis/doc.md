# Analysis Guide

This directory contains the project’s analysis-oriented code: loading the curated AAT dataset, computing summary views over it, and producing a standalone HTML dashboard that can be opened in a browser.

## What this part of the project does

The analysis side is meant to answer questions such as:

- How are AAT terms distributed across the chosen museum-relevant hierarchies?
- Which centuries appear most often in the descriptive notes?
- How deep or shallow are different parts of the taxonomy?
- How much multilingual coverage exists in the filtered subset?

The result is not a prediction system. It is a way to understand the structure and coverage of the vocabulary the museum workflow depends on.

## Main files

### `load_dataset.py`

This is the entry point for getting analysis data into a usable form.

What it does:

- Looks for a local parquet file in `data_cache/`.
- If the parquet file already exists, loads it immediately.
- If it does not exist, downloads the dataset from Hugging Face using the `datasets` library.
- Saves the downloaded table as a parquet file for future runs.

Why it matters:

- It keeps the analysis workflow fast after the first download.
- It avoids repeated network calls when someone is just regenerating charts.
- It defines the default dataset used by the dashboard scripts.

### `09_dashboard.py`

This is the standalone dashboard generator.

What it does:

- Reads the parquet dataset from `data_cache/` by default.
- Computes chart inputs from columns such as `preferred_term`, `hierarchy`, `scope_note`, `parent_id`, and `variant_terms`.
- Builds Plotly figures for facet balance, century mentions, parent-term structure, keyword trends, tree depth, and geographic/language patterns.
- Writes a self-contained HTML file to `figures/dashboard.html`.

Important detail:

- This file creates a browser-openable HTML dashboard.
- It does not run a web server and it is not the same thing as the Gradio exhibit in `src/frontend/data_viz/`.

## Folder contents

### `data_cache/`

This folder stores cached parquet snapshots used by the analysis code.

The main file is:

- `aat_museum_subset.parquet`

That dataset is the museum-focused subset of the Getty AAT that the rest of the analysis work depends on. It contains one row per term, with fields such as:

- `preferred_term`
- `subject_id`
- `hierarchy`
- `parent_id`
- `parent_term`
- `scope_note`
- `variant_terms`

This folder mostly contains data artifacts, not hand-written source code.

### `figures/`

This folder stores output from the analysis process.

The most important file is:

- `dashboard.html`

That file is the main generated deliverable from `09_dashboard.py`.

The folder also contains several static PNG figures from earlier exploratory work. Those are useful references, but they are not the primary current output and are not all regenerated by the current script.

## How the analysis flow works

Typical flow:

1. Run `load_dataset.py` or let another script trigger it implicitly.
2. Read the cached parquet dataset from `data_cache/`.
3. Compute derived values such as tree depth, scope-note length, or counts by hierarchy.
4. Build Plotly figures.
5. Save the combined dashboard HTML into `figures/dashboard.html`.

## Relationship to the frontend exhibit

The analysis code and the exhibit code use the same dataset family, but they are separate products.

- `src/analysis/09_dashboard.py` creates a standalone HTML dashboard.
- `src/frontend/data_viz/data_story_exhibit.py` creates a Gradio-based interactive exhibit.

They overlap in subject matter, but not in implementation.

## Good places to look when editing this area

- Edit `load_dataset.py` if the source dataset, caching behavior, or parquet path changes.
- Edit `09_dashboard.py` if the story, charts, or output HTML should change.
- Check `data_cache/` if results look wrong because the input snapshot may be stale.
- Check `figures/` if someone is asking about generated outputs rather than source code.
